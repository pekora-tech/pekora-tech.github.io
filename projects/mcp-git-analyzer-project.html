專案背景與目標

核心問題識別

作為系統分析師，在理解新專案時經常面臨的核心挑戰：

  • 歷程理解困難：從Git歷史中快速掌握專案發展脈絡
  • 技術決策追蹤：識別關鍵的架構演進和技術選型決策
  • 階段劃分模糊：缺乏系統性的專案開發階段識別方法
  • 重複性工作：手動分析Git歷史的效率低下

第一性原理分析

專案理解本質 = 時間序列分析 + 變更模式識別 + 決策點提取
自動化核心 = 模式匹配算法 + 結構化輸出 + 可配置參數
工具價值 = 效率提升 × 準確性保證 × 知識積累

專案目標設定

  • 主要目標：建立SA專用的Git專案理解工具
  • 效率目標：將專案初步理解時間從4小時縮短至30分鐘
  • 品質目標：實現90%的關鍵技術決策點自動識別
  • 擴展目標：建立可複製的MCP工具開發模式

技術選型決策

Node.js + MCP協議

選擇理由：

  • MCP生態系統與Claude/Gemini的原生整合
  • JavaScript的JSON處理能力
  • npm生態系統的豐富性
  • 跨平台部署支援

替代方案考量：

  • Python：更強的資料分析能力，但MCP整合複雜度高
  • Go：效能優勢，但開發效率較低
  • Rust：最佳效能，但學習曲線陡峭

simple-git庫選擇

選擇理由：

  • 純JavaScript實現，無原生依賴
  • Promise支援，適合async/await模式
  • 完整的Git命令封裝
  • 活躍維護，社群支援良好

效能考量：

  • 針對大型專案（1000+ commits）進行分頁處理
  • 實現增量分析，避免重複計算
  • 快取機制減少Git操作次數

技術債務管理

已知限制

算法精度限制

  • commit分類精度：基於正則表達式，語義理解有限
  • 階段檢測準確性：依賴模式匹配，複雜專案可能失效
  • 決策點識別：可能遺漏隱含的技術決策

效能約束

  • 大型專案處理：超過10000個commit時效能下降
  • 記憶體使用：批次處理時可能出現記憶體尖峰
  • 並行處理：當前為序列處理，未充分利用多核心

相容性問題

  • Git版本相容性：部分高級功能需要Git 2.23+
  • 平台差異：Windows路徑處理與Unix系統有差異
  • Node.js版本依賴：ES模組特性需要Node.js 14+
