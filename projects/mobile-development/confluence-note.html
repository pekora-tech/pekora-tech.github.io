背景與問題

核心理念

將「人事時地物」以及核心的「事」（動作/動詞）作為組織筆記的核心維度，構建一個非常強大且貼合軟體工程師思維的工具。

設計哲學

  • 以「事」為中心：筆記的核心是發生的「事件」或「動作」
  • 五維度關聯：每個「事」都可以關聯到具體的「人、地、物、時」
  • 高度結構化：不是自由文本，而是結構化的數據關聯
  • 軟體工程思維：OOP概念，萬物皆可當成物品/對象

技術方案

🎯 核心概念實體化

1. 地 (Place) - 物理或虛擬空間

2. 人 (Person) - 個體或群體

  • 分三區設計：稱謂、人分類、人資料
  • 引用分類稱謂跟地方資料Set
  • 包含人名、分機、手機、備註等

3. 物 (WorkObject) - 工作對象、項目或產出物

  • OOP思維：萬物皆可當成物品
  • 軟體工程師的項目拆解概念

4. 事 (ActionVerb) - 核心動作、事件或主題

  • 核心動詞，描述「做什麼」
  • 補足物的不可具象性
  • 可能非常抽象的動作概念

5. 時 (NoteEntry) - 基於前四者建立的筆記條目

  • 利用1~4的資源建置筆記
  • 時間維度的具體體現
  • 最終的知識載體

🏗️ 系統架構設計

左側導航欄結構

主視圖
├── 總覽/儀表板
└── 所有筆記

按「事」核心動詞/主題
├── 討論相關
├── 設計工作  
└── 開發日誌

按「物」(工作項/專案)
├── 按類型 (會議、開發任務、文檔)
└── 按專案

按「人」
├── 按分類 (同事、客戶、合作夥伴)
└── 所有人列表

按「地」
├── 會議室
├── 辦公室
└── 遠程會議

按「時」
├── 今天/本週
└── 日曆視圖

數據管理/設定
├── 管理地點
├── 管理人事資料
├── 管理工作項 (物)
├── 管理動作庫 (事)
└── 應用設定

技術選型

  • 前端框架: Flutter (跨平台支持)
  • 數據存儲: SQLite (本地數據庫)
  • 認證方式: 本地認證
  • ID 生成: UUID 套件
  • 用戶設置: SharedPreferences

核心實作

數據模型關聯設計

每個筆記(NoteEntry)可以關聯：

  • 多個人員 (personIds)
  • 一個主要地點 (primaryPlaceId)
  • 多個工作項 (workObjectIds)
  • 一個核心動作 (coreActionVerb)

創新的UI/UX設計

  • 左右分欄佈局：左側導航，右側內容
  • 維度化導航：按五個維度組織信息
  • 結構化表單：引導用戶建立關聯
  • 多維度篩選：支援複合查詢條件

企業級架構模式

  • Repository Pattern：數據存取層抽象
  • MVVM架構：清晰的層次分離
  • 依賴注入：使用Provider管理狀態
  • 單一職責：每個模塊專注特定功能
